Experimental Analysis of the T (Plain Changes) Algorithm

  • CT-208 Computer Mathematics (ITA) - Prof. Luiz Mirizola
  • Authors: Flavio Souza and Nilson Sandy

Unlike the classic lexicographic algorithm, which may require rearranging large blocks of the vector, Algorithm T focuses on the simplicity of the movement. Its premise is elegant: each new permutation is generated from the previous one through a single adjacent change (swapping two neighboring elements).

This characteristic defines it as a "minimal change" algorithm, ideal for systems where the cost of changing the state must be kept to a minimum.

Didactic Example (Online)

-
At each step, only one adjacent pair changes.

How Algorithm T Works

Algorithm T is split into two simple functions. The offline phase builds a swap plan once. The online phase reuses that plan to generate permutations quickly.

Online T - Pseudocode

        T_ONLINE(vector):
          # Goal: generate permutations directly, finding the mobile element at each step.
          a <- copy(vector)
          d <- [-1, ..., -1]
          output a

          while true:
            # Block 1: search the largest mobile element and its index.
            mobile <- -1
            idx_mobile <- -1

            for i in 0..n-1:
              prox <- i + d[i]
              if 0 <= prox < n and a[i] > a[prox] and a[i] > mobile:
                mobile <- a[i]
                idx_mobile <- i

            # Block 2: stop condition when no mobile element exists.
            if idx_mobile == -1: break

            # Block 3: perform the adjacent swap and keep direction aligned.
            destination <- idx_mobile + d[idx_mobile]
            record (idx_mobile, destination)
            swap a[idx_mobile], a[destination]
            swap d[idx_mobile], d[destination]

            # Block 4: reverse direction of elements larger than the moved mobile.
            for i in 0..n-1:
              if a[i] > mobile:
                d[i] <- -d[i]

            output a

Offline T - Pseudocode

        TRANSACTION_TABLE(n):
          # Goal: precompute only the swap indices (transaction list), not full permutations.
          run the same mobile-element loop above over a <- [1..n]
          store only each (idx_mobile, prox) in transactions
          return transactions

        T_OFFLINE(vector):
          # Goal: execute fast using the precomputed transaction list.
          transactions <- TRANSACTION_TABLE(n)
          a <- copy(vector)
          output a

          # Block: replay each adjacent swap to enumerate all permutations.
          for each (i, j) in transactions:
            swap a[i], a[j]
            output a

Offline Version

This version prepares the swap plan. It finds which adjacent positions must be swapped to move from one permutation to the next. The result is a compact roadmap.

Online Version

This version receives the precomputed plan and applies each swap in order. It does not recompute the logic of the next move; it only follows the plan.

Step by Step

1. Start with the identity permutation.
2. Build the swap plan offline.
3. Reuse that plan online to generate every new permutation.
4. Measure offline and online costs separately.
5. Compare T with P.
